## 1. 引言

### 1.1 文档目的

本文档对 SalesPilot 系统的 MongoDB 数据库进行完整设计，涵盖：集合（Collection）结构定义、字段规范、索引策略、实体关系、数据字典、多租户隔离方案、容量规划及运维策略。

本文档作为数据库实施的权威依据，开发、测试、运维人员均应遵照执行。

### 1.2 设计目标

| 目标         | 具体指标                                              |
| ------------ | ----------------------------------------------------- |
| 查询性能     | 常用列表查询 P90 响应时间 < 100ms                     |
| 数据一致性   | 关键业务操作（成交、删除）采用事务保证数据一致性      |
| 数据安全     | 多租户严格隔离，A 公司不可访问 B 公司任何数据         |
| 可扩展性     | Schema 预留扩展字段，支持平滑迭代，不破坏现有数据     |
| 存储效率     | MVP 阶段 Atlas M0（512MB）满足 6 个月运营需求         |

### 1.3 技术选型说明

选择 MongoDB 而非关系型数据库（MySQL / PostgreSQL）的核心理由：

| 维度         | MongoDB                                      | MySQL                                    |
| ------------ | -------------------------------------------- | ---------------------------------------- |
| 跟进记录     | `content` 为非结构化长文本，文档模型天然适配 | 需要单独 TEXT 字段，无法灵活扩展         |
| AI 分析结果  | `positives[]` / `risks[]` 数组字段自然存储   | 需要额外关联表，增加 JOIN 复杂度         |
| Schema 演进  | 无迁移成本，字段可随版本迭代增减             | 每次变更需要 ALTER TABLE，有锁表风险     |
| 聚合查询     | Aggregation Pipeline 表达力强，适合统计大盘  | 复杂 GROUP BY 语句可读性差               |
| 部署成本     | Atlas M0 免费，开箱即用                      | 需要自购云数据库，有固定成本             |

### 1.4 阅读约定

- `[PK]` — 主键（MongoDB 中为 `_id`，ObjectId 类型）
- `[FK]` — 外键引用（MongoDB 中为存储 ObjectId 字符串，逻辑引用）
- `[idx]` — 建有单字段索引
- `[cidx]` — 参与复合索引
- `[unique]` — 唯一约束
- `[required]` — 必填字段，不可为 null 或 undefined
- `[ttl]` — 设置 TTL 自动过期索引
- 字段类型说明：`String` / `Number` / `Boolean` / `Date` / `ObjectId` / `Array<T>` / `Mixed`

## 2. 数据库总体设计

### 2.1 数据库命名规范

| 环境       | 数据库名                | 说明                     |
| ---------- | ----------------------- | ------------------------ |
| 开发       | `salespilot_dev`        | 本地开发，允许随意操作   |
| 测试       | `salespilot_test`       | 自动化测试，每次运行重置 |
| 预发布     | `salespilot_staging`    | 与生产结构完全一致       |
| 生产       | `salespilot_prod`       | 禁止直接写入，只读账号   |

### 2.2 集合命名规范

- 全部小写，单词间用下划线 `_` 分隔
- 使用复数形式（`users`，而非 `user`）
- 禁止使用 MongoDB 保留集合名（`system.*`）

### 2.3 字段命名规范

- 统一使用 **camelCase**（驼峰命名）
- 布尔字段以 `is` / `has` 开头（`isDeleted`、`isRead`、`isActive`）
- 时间字段以 `At` 结尾（`createdAt`、`updatedAt`、`lastLoginAt`）
- 外键引用字段以 `Id` 结尾（`companyId`、`assignedTo` 除外为历史设计）

### 2.4 通用字段规范

所有集合必须包含以下通用字段（通过 Mongoose `timestamps: true` 自动管理）：

| 字段名      | 类型    | 说明                            |
| ----------- | ------- | ------------------------------- |
| `_id`       | ObjectId | 主键，MongoDB 自动生成         |
| `createdAt` | Date    | 创建时间，自动填充              |
| `updatedAt` | Date    | 最后更新时间，自动维护          |

除 `companies` 外，所有集合还必须包含：

| 字段名      | 类型    | 说明                                            |
| ----------- | ------- | ----------------------------------------------- |
| `companyId` | ObjectId | 租户 ID，所有查询必须携带，确保数据隔离         |

## 3. 集合设计总览

### 3.1 集合列表

| 集合名          | 中文名           | 用途说明                          | 预计文档量（MVP 6个月） |
| --------------- | ---------------- | --------------------------------- | ----------------------- |
| `companies`     | 公司/租户        | SaaS 多租户的顶级隔离单元         | < 100                   |
| `users`         | 用户             | 销售员、主管、管理员账号          | < 1,000                 |
| `customers`     | 客户             | 客户主数据，销售管理核心          | < 10,000                |
| `followups`     | 跟进记录         | 客户跟进流水，AI 分析的数据来源   | < 200,000               |
| `ai_results`    | AI 分析结果      | 成交预测、流失预警的缓存结果      | < 20,000                |
| `notifications` | 通知/预警        | 流失预警、系统通知               | < 50,000                |

### 3.2 集合关系概览

```
companies (1)
    └── users (N)           一家公司有多个用户
    └── customers (N)       一家公司有多个客户
         └── followups (N)  一个客户有多条跟进记录
         └── ai_results (N) 一个客户有多次 AI 分析
    └── notifications (N)   一家公司有多条通知
```

## 4. 集合详细设计

### 4.1 companies（公司/租户）

#### 4.1.1 集合说明

多租户 SaaS 的顶级隔离单元。每个注册公司对应一条记录，所有其他集合通过 `companyId` 关联至此。

#### 4.1.2 字段定义

| 字段名        | 类型    | 约束                   | 默认值    | 说明                                    |
| ------------- | ------- | ---------------------- | --------- | --------------------------------------- |
| `_id`         | ObjectId | `[PK]`                |           | 主键                                    |
| `name`        | String  | `[required]` 2-100字   |           | 公司名称                                |
| `slug`        | String  | `[unique]`             |           | URL 友好标识，自动生成（预留）          |
| `plan`        | String  | enum                   | `'free'`  | 订阅套餐：`free` / `pro` / `enterprise` |
| `maxUsers`    | Number  | min: 1                 | `5`       | 套餐允许的最大用户数                    |
| `maxCustomers`| Number  | min: 1                 | `100`     | 套餐允许的最大客户数                    |
| `aiCallsUsed` | Number  | min: 0                 | `0`       | 今日已使用 AI 调用次数（每日重置）      |
| `aiCallsDate` | Date    |                        | `null`    | aiCallsUsed 的统计日期                  |
| `isActive`    | Boolean |                        | `true`    | 是否启用（停用后所有用户无法登录）      |
| `settings`    | Mixed   |                        | `{}`      | 扩展配置（预留）                        |
| `createdAt`   | Date    | 自动                   |           | 创建时间                                |
| `updatedAt`   | Date    | 自动                   |           | 更新时间                                |

#### 4.1.3 索引

| 索引名               | 字段            | 类型     | 说明                     |
| -------------------- | --------------- | -------- | ------------------------ |
| `_id_`               | `_id`           | 主键索引 | MongoDB 自动创建         |
| `companies_slug_idx` | `slug: 1`       | 唯一索引 | 快速按 slug 查找公司     |

#### 4.1.4 示例文档

```json

  "plan": "free",
  "maxUsers": 5,
  "maxCustomers": 100,
  "aiCallsUsed": 12,
  "aiCallsDate": "2026-03-19T00:00:00.000Z",
  "isActive": true,
  "settings": {},
  "createdAt": "2026-03-01T08:00:00.000Z",
  "updatedAt": "2026-03-19T09:00:00.000Z"
}
```

#### 4.2.1 集合说明

存储系统所有用户信息，包含登录凭证、角色权限和公司归属。密码字段使用 bcrypt 哈希存储，默认查询不返回。

#### 4.2.2 字段定义

| 字段名        | 类型    | 约束                       | 默认值    | 说明                                              |
| ------------- | ------- | -------------------------- | --------- | ------------------------------------------------- |
| `_id`         | ObjectId | `[PK]`                    |           | 主键                                              |
| `companyId`   | ObjectId | `[FK→companies]` `[required]` `[cidx]` |  | 所属公司                            |

#### 4.3.4 status 枚举值与流转规则

```
潜在客户(potential) ──→ 跟进中(following) ──→ 意向确认(intent)
                                                      │
                              ┌──────────────────────▼──────────────┐
                              │         报价谈判(negotiating)         │
                              └──────────────────────┬──────────────┘
                                                     │
                              ┌──────────────────────┼──────────────┐
                              ▼                                     ▼
                          已成交(won)                           已流失(lost)
```

| 状态值        | 显示名称 | 说明                             | 允许流转至                          |
| ------------- | -------- | -------------------------------- | ----------------------------------- |
| `potential`   | 潜在客户 | 新录入，尚未开始跟进             | `following`、`lost`                 |
| `following`   | 跟进中   | 已有跟进记录，主动推进中         | `intent`、`lost`                    |
| `intent`      | 意向确认 | 客户明确表达购买意向             | `negotiating`、`lost`               |
| `negotiating` | 报价谈判 | 正在讨论价格或合同条款           | `won`、`lost`、`intent`             |
| `won`         | 已成交   | 合同签订，项目落地               | 不允许流转（终态）                  |
| `lost`        | 已流失   | 明确拒绝或长期无响应             | `following`（重新激活）             |

| 枚举值        | 显示名称 |
| ------------- | -------- |
| `website`     | 官网/SEO |
| `referral`    | 转介绍   |

| `cold_call`   | 电话开发 |
| `social`      | 社交媒体 |

#### 4.3.6 索引

| 索引名                          | 字段                                          | 类型     | 查询场景                          |

### 4.4 followups（跟进记录）

#### 4.4.1 集合说明

记录销售员与客户的每次互动流水。`content`（跟进内容）是 AI 成交预测和日报生成的**核心数据来源**，应引导用户尽量详细填写。

`companyId` 为冗余字段（可从 `customerId` 关联推导），但为避免聚合查询时的 `$lookup`，直接存储以提升日报统计性能。

#### 4.4.2 字段定义

| 字段名        | 类型    | 约束                              | 默认值 | 说明                                                  |

| `_id`         | ObjectId | `[PK]`                           |        | 主键                                                  |
| `companyId`   | ObjectId | `[FK→companies]` `[required]` `[cidx]` |   | 所属公司（冗余字段，加速聚合）                        |
| `customerId`  | ObjectId | `[FK→customers]` `[required]` `[cidx]` |   | 关联客户                                              |

### 4.5 ai_results（AI 分析结果）

#### 4.5.1 集合说明

缓存 AI 大模型对客户的分析结果，避免重复调用 API（默认 12 小时缓存）。同一客户可存在多条历史记录，按时间排序取最新一条作为当前结果。

`rawPrompt` 字段仅在开发/测试环境存储，用于 Prompt 调试和优化，生产环境通过配置关闭以节省存储。

#### 4.5.2 字段定义

| 字段名      | 类型           | 约束                             | 默认值      | 说明                                                     |
| ----------- | -------------- | -------------------------------- | ----------- | -------------------------------------------------------- |
| `_id`       | ObjectId       | `[PK]`                           |             | 主键                                                     |
| `companyId` | ObjectId       | `[FK→companies]` `[required]` `[cidx]` |       | 所属公司                                                 |
| `customerId`| ObjectId       | `[FK→customers]` `[required]` `[cidx]` |       | 关联客户                                                 |
| `type`      | String         | `[required]` enum                |             | 分析类型：`deal_prediction`（成交预测）/`churn_warning`（流失预警） |
| `score`     | Number         | min:-1, max:100                  | `-1`        | 成交概率 0-100，`-1` 表示无评分（流失预警类型）           |
| `level`     | String         | enum                             | `'unknown'` | 等级：`high`/`medium`/`low`/`unknown`                    |
| `positives` | Array\<String\>|                                  | `[]`        | 正向信号列表，AI 输出，≤ 3 条                             |
| `risks`     | Array\<String\>|                                  | `[]`        | 风险点列表，AI 输出，≤ 3 条                               |
| `suggestion`| String         | 最大 500 字                      | `''`        | AI 建议的下一步跟进动作                                   |
| `modelUsed` | String         |                                  | `''`        | 使用的模型名称（如 `qwen-plus`），便于 A/B 测试           |
| `rawPrompt` | String         | select:false                     | `null`      | 完整 Prompt，**默认不查询返回**，生产环境建议不存         |
| `createdAt` | Date           | 自动 `[idx]`                     |             | 分析时间，缓存有效期判断依据                              |
| `updatedAt` | Date           | 自动                             |             | 更新时间                                                 |

#### 4.5.3 缓存策略

```
触发分析：
  新增跟进记录 → 后台异步触发（不阻塞主流程）
  用户手动点击「重新分析」→ 强制触发（忽略缓存）

缓存判断逻辑：
  查询条件：{ customerId, type, createdAt: { $gte: now - 12h } }
  命中缓存 → 直接返回，不调用 AI API
  未命中   → 调用 AI API → 写入新记录

历史记录保留策略：
  每个客户每种类型保留最近 10 条历史（定期清理旧记录）
  便于查看分析结果的历史变化趋势
```

#### 4.5.4 索引

| 索引名                     | 字段                                        | 类型 | 查询场景                            |
| -------------------------- | ------------------------------------------- | ---- | ----------------------------------- |
| `_id_`                     | `_id`                                       | 主键 | MongoDB 自动                        |
| `ai_results_cache_idx`     | `customerId:1, type:1, createdAt:-1`        | 复合 | 缓存查询（最高频，每次分析前检查）  |

## 5. 索引设计

### 5.1 索引汇总

| 集合            | 索引名                             | 字段                                              | 类型     | 用途                        |
| --------------- | ---------------------------------- | ------------------------------------------------- | -------- | --------------------------- |
| `companies`     | `_id_`                             | `_id`                                             | 主键     | 默认                        |
| `companies`     | `companies_slug_idx`               | `slug:1`                                          | 唯一     | slug 查找                   |
| `users`         | `_id_`                             | `_id`                                             | 主键     | 默认                        |
| `users`         | `users_email_idx`                  | `email:1`                                         | 唯一     | 登录认证                    |
| `users`         | `users_company_role_idx`           | `companyId:1, role:1`                             | 复合     | 公司用户列表                |
| `customers`     | `_id_`                             | `_id`                                             | 主键     | 默认                        |
| `customers`     | `customers_list_idx`               | `companyId:1, isDeleted:1, createdAt:-1`          | 复合     | 主列表分页                  |
| `customers`     | `customers_status_assigned_idx`    | `companyId:1, status:1, assignedTo:1`             | 复合     | 看板 + 筛选                 |
| `customers`     | `customers_ai_score_idx`           | `companyId:1, aiScore:-1`                         | 复合     | AI 评分排序                 |
| `customers`     | `customers_churn_scan_idx`         | `companyId:1, status:1, lastFollowup:1`           | 复合     | 流失预警扫描                |
| `followups`     | `_id_`                             | `_id`                                             | 主键     | 默认                        |
| `followups`     | `followups_timeline_idx`           | `customerId:1, createdAt:-1`                      | 复合     | 客户时间线                  |
| `followups`     | `followups_daily_idx`              | `companyId:1, createdBy:1, createdAt:-1`          | 复合     | 日报统计                    |
| `followups`     | `followups_customer_count_idx`     | `customerId:1, createdAt:1`                       | 复合     | AI 频率统计                 |
| `ai_results`    | `_id_`                             | `_id`                                             | 主键     | 默认                        |
| `ai_results`    | `ai_results_cache_idx`             | `customerId:1, type:1, createdAt:-1`              | 复合     | 缓存查询                    |
| `ai_results`    | `ai_results_company_idx`           | `companyId:1, type:1, createdAt:-1`               | 复合     | 统计分析                    |
| `notifications` | `_id_`                             | `_id`                                             | 主键     | 默认                        |
| `notifications` | `notifications_inbox_idx`          | `companyId:1, targetUserId:1, isRead:1, createdAt:-1` | 复合 | 收件箱                  |
| `notifications` | `notifications_company_idx`        | `companyId:1, type:1, createdAt:-1`               | 复合     | 公司统计                    |

**总计：19 个索引**（含主键索引 6 个，业务索引 13 个）

### 5.2 索引设计原则

**原则一：companyId 始终置于复合索引第一位**

MongoDB 使用最左前缀匹配规则。所有涉及公司数据的查询均携带 `companyId`，将其置首可最大化索引利用率，同时保障租户数据隔离。

```
✅ { companyId:1, status:1, assignedTo:1 }
❌ { status:1, companyId:1, assignedTo:1 }  // companyId 不在首位，性能差
```

**原则二：范围查询字段置于等值查询字段之后**

```
✅ { companyId:1, isDeleted:1, createdAt:-1 }
   // companyId（等值）→ isDeleted（等值）→ createdAt（范围排序）
```

**原则三：高写入集合控制索引数量**

`followups` 为高写入集合（每次跟进写一条），索引过多会拖慢写入性能。当前设计 3 个业务索引（不含主键），保持在合理范围。

**原则四：定期执行 explain() 验证索引使用**

```javascript
// 验证主列表查询是否使用索引（IXSCAN 为命中，COLLSCAN 为全表扫描）
db.customers.find({
  companyId: ObjectId("..."),
  isDeleted: false
}).sort({ createdAt: -1 }).explain("executionStats")
```

## 6. 实体关系图（ER Diagram）

### 6.1 文字描述版（E-R 关系）

```
companies ──┬── users (1:N)
            │     └── 一家公司可有多个用户（admin/manager/sales）
            │
            ├── customers (1:N)
            │     ├── 一家公司可有多个客户
            │     ├── 每个客户有一个负责销售员（assignedTo → users）
            │     │
            │     ├── followups (1:N)
            │     │     └── 一个客户有多条跟进记录
            │     │           └── createdBy → users（操作人）
            │     │
            │     └── ai_results (1:N)
            │           └── 一个客户有多次 AI 分析历史
            │
            └── notifications (1:N)
                  ├── 一家公司有多条通知
                  ├── targetUserId → users（接收人，null=全员）
                  └── relatedId → customers（关联客户，可为null）
```

### 6.2 关系说明

| 关系                              | 类型  | 说明                                                    |
| --------------------------------- | ----- | ------------------------------------------------------- |
| `companies` → `users`             | 1 : N | 一个公司拥有多个用户                                    |
| `companies` → `customers`         | 1 : N | 一个公司拥有多个客户                                    |
| `users` → `customers`             | 1 : N | 一个销售员负责多个客户（`assignedTo` 字段）             |
| `customers` → `followups`         | 1 : N | 一个客户有多条跟进记录                                  |
| `users` → `followups`             | 1 : N | 一个用户创建多条跟进记录（`createdBy` 字段）            |
| `customers` → `ai_results`        | 1 : N | 一个客户有多次 AI 分析（历史记录）                      |
| `companies` → `notifications`     | 1 : N | 一个公司有多条通知/预警                                 |
| `customers` → `notifications`     | 1 : N | 一个客户可关联多条预警通知                              |

### 6.3 字段引用关系

```
users.companyId          ──→  companies._id
customers.companyId      ──→  companies._id
customers.assignedTo     ──→  users._id
followups.companyId      ──→  companies._id
followups.customerId     ──→  customers._id
followups.createdBy      ──→  users._id
ai_results.companyId     ──→  companies._id
ai_results.customerId    ──→  customers._id
notifications.companyId  ──→  companies._id
notifications.targetUserId ─→ users._id（可为 null）
notifications.relatedId  ──→  customers._id（可为 null）
```

> **说明**：MongoDB 不强制外键约束，所有引用关系由应用层代码保证。删除操作（如软删除客户）时，关联数据（followups、ai_results）保留，不级联删除。

## 8. 多租户数据隔离策略

### 8.1 隔离方案选型

系统采用**共享数据库 + 字段隔离**（Shared Database, Separate Collections with companyId filter）方案：

| 方案                          | 隔离级别 | 成本   | 适用场景                    | SalesPilot 选择 |
| ----------------------------- | -------- | ------ | --------------------------- | --------------- |
| 每租户独立数据库              | 最强     | 最高   | 金融、政府等高安全需求      | ❌              |
| 共享数据库 + 独立 Collection  | 强       | 较高   | 大型 SaaS，数据量差异大     | ❌              |
| 共享数据库 + companyId 字段   | 中       | 最低   | 中小 SaaS，MVP 阶段         | ✅              |

### 8.2 隔离实施规范

**规范一：所有数据查询必须携带 companyId**

```javascript
// ✅ 正确：Service 层注入 companyId
const customers = await Customer.find({
  companyId: req.user.companyId,   // 从 JWT 中读取，不信任前端传入
  isDeleted: false,
  ...otherFilters,
});

// ❌ 错误：遗漏 companyId，导致跨租户数据泄露
const customers = await Customer.find({ status: 'following' });
```

**规范二：companyId 从 JWT 读取，不接受客户端传入**

```javascript
// ✅ 正确：从 JWT payload 获取
const companyId = req.user.companyId;

// ❌ 错误：信任客户端传入的 companyId
const companyId = req.body.companyId;  // 可被篡改，安全漏洞
```

**规范三：写入操作同样必须携带 companyId 过滤**

```javascript
// ✅ 正确：更新时验证 companyId 归属
const result = await Customer.updateOne(
  { _id: customerId, companyId: req.user.companyId },  // 双重验证
  { $set: { status: newStatus } }
);
if (result.matchedCount === 0) throw new AppError('CUSTOMER_NOT_FOUND', 404);

// ❌ 错误：仅凭 _id 更新，可能修改他公司数据
await Customer.updateOne({ _id: customerId }, { $set: { status } });
```

**规范四：删除操作使用软删除 + companyId 验证**

```javascript
// ✅ 正确：软删除，带公司验证
await Customer.updateOne(
  { _id: id, companyId: req.user.companyId },
  { $set: { isDeleted: true } }
);
```

### 8.3 集成测试验证

每次 PR 合并前，必须通过以下跨租户隔离测试：

| 测试场景                           | 期望结果                                     |
| ---------------------------------- | -------------------------------------------- |
| A 公司 Token 查询所有客户          | 仅返回 A 公司客户，不含 B 公司数据           |
| A 公司 Token 更新 B 公司客户       | 返回 404，客户不存在                         |
| A 公司 Token 删除 B 公司客户       | 返回 404，客户不存在                         |
| A 公司 Token 查看 B 公司 AI 结果   | 返回空列表或 404                             |
| 构造 JWT 伪造 companyId            | 中间件校验失败，返回 401                     |

### 9.2 notifications TTL 索引配置

```javascript
// notifications 集合：90 天后自动删除
NotificationSchema.index(
  { createdAt: 1 },
  { expireAfterSeconds: 90 * 24 * 3600 }  // 7,776,000 秒
```

### 9.3 ai_results 历史清理任务

```javascript
// 每周日 03:00 执行，保留每客户每类型最近 10 条
cron.schedule('0 3 * * 0', async () => {
  const pipeline = [
    { $group: { _id: { customerId: '$customerId', type: '$type' },
                docs: { $push: { id: '$_id', createdAt: '$createdAt' } } } },
    { $project: {
        toDelete: {
          $slice: [
            { $sortArray: { input: '$docs', sortBy: { createdAt: -1 } } },
            10,       // 跳过最新 10 条
            9999,     // 取剩余所有
          ]
        }
      }
    },
  ];

  const groups = await AiResult.aggregate(pipeline);
  const idsToDelete = groups.flatMap(g => g.toDelete.map(d => d.id));

  if (idsToDelete.length > 0) {
    await AiResult.deleteMany({ _id: { $in: idsToDelete } });
    logger.info(`[CleanJob] 清理旧 AI 结果 ${idsToDelete.length} 条`);
  }
});
```

## 11. 备份与恢复策略

### 11.1 Atlas 自动备份

MongoDB Atlas 提供以下备份能力（M10 及以上）：

| 备份类型     | 频率       | 保留时间 | 操作方式              |
| ------------ | ---------- | -------- | --------------------- |
| 连续备份     | 实时       | 24 小时  | Atlas 控制台一键恢复  |
| 每日快照     | 每天       | 7 天     | 指定时间点恢复        |
| 每周快照     | 每周       | 4 周     | 指定快照恢复          |

> M0 免费层不支持自动备份，需手动导出或升级至 M2+。

### 11.2 MVP 阶段手动备份方案

M0 免费层使用 `mongodump` 定期导出：

```bash
# 每日备份脚本（部署在后端服务器，通过 cron 执行）
#!/bin/bash
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/backups/salespilot_${DATE}"

mongodump \
  --uri="${MONGODB_URI}" \
  --out="${BACKUP_DIR}" \
  --gzip

# 压缩打包
tar -czf "${BACKUP_DIR}.tar.gz" "${BACKUP_DIR}"
rm -rf "${BACKUP_DIR}"

# 保留最近 7 天备份
find /backups -name "*.tar.gz" -mtime +7 -delete

echo "[Backup] 完成：${BACKUP_DIR}.tar.gz"
```

### 11.3 数据恢复流程

```
1. 确认恢复时间点
2. 停止应用服务（避免写入冲突）
3. 执行恢复：
   mongorestore --uri="${MONGODB_URI}" --gzip --drop ./backup_dir/
4. 验证数据完整性（文档数量、关键字段）
5. 重启应用服务
6. 通知相关人员恢复完成
```

```typescript
import mongoose from 'mongoose';

  const uri = process.env.MONGODB_URI!;
```
